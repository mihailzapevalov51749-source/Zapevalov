# YASNOPRO VIEW ENGINE MODEL

## 1. Назначение документа

Документ описывает модель View Engine платформы ЯсноПро.

### Цель документа

- определить, как платформа отображает данные пользователям;
- разделить данные, представления, настройки и блоки;
- устранить путаницу между View, Representation, Tab и Block;
- создать основу для стабильной Universal Table;
- подготовить фундамент для будущих видов отображения: Kanban, Gantt, Calendar, Composite View.

### Простыми словами

Этот документ отвечает на вопрос:

«Как ЯсноПро показывает объекты компании людям?»

## 2. Главная идея View Engine

View Engine — это движок отображения данных.

### По-человечески

View Engine — это система, которая берёт объекты компании и показывает их пользователю в удобном виде.

Например:

- как таблицу;
- как дерево;
- как канбан;
- как календарь;
- как график;
- как карточку;
- как составной экран.

## 3. Что View Engine НЕ делает

View Engine НЕ является:

- источником данных;
- хранилищем бизнес-объектов;
- заменой Entity Layer;
- заменой ObjectType;
- местом бизнес-логики.

View Engine только отображает данные.

## 4. Главный принцип

Entity Layer хранит данные.

View Engine показывает данные.

Representation хранит настройки отображения.

Block размещает View на странице.

## 5. Основные понятия

| Термин | Русское понимание |
|---|---|
| Entity | сам объект компании |
| View | способ посмотреть на объект |
| View Type | тип отображения |
| Representation | настройки конкретного отображения |
| View Session | временное состояние открытого отображения |
| Block | место на странице, куда вставлен View |
| Layout | расположение блоков |

## 6. Что такое View

### Технически

View — способ отображения Entity или набора Entity.

### По-человечески

View — это способ посмотреть на данные.

Например:

- список проектов таблицей;
- задачи по статусам в канбане;
- сроки проектов на календаре;
- структура компании деревом.

## 7. View НЕ хранит данные

View не должен быть source of truth.

Если пользователь видит проект в таблице, проект не принадлежит таблице.

Проект принадлежит Entity Layer.

Таблица только показывает проект.

## 8. Что такое View Type

### Технически

View Type — тип отображения данных.

### По-человечески

View Type — это формат, в котором человек смотрит на данные.

## 9. Основные View Types

| View Type | Русское понимание | Для чего нужен |
|---|---|---|
| Table | Таблица | Реестры, списки, данные по столбцам |
| Tree | Дерево | Иерархия, структура, вложенность |
| Kanban | Канбан | Работа по статусам |
| Gantt | Гант | План-график и сроки |
| Calendar | Календарь | События и даты |
| Card | Карточка | Детальный просмотр объекта |
| Composite | Составное представление | Несколько View в одном экране |
| Dashboard | Панель управления | Аналитика и показатели |

## 10. Вкладки View

Вкладки вида:

- Таблица;
- Дерево;
- Составное;
- Канбан;
- Календарь.

Это НЕ Representation.

Это переключение между View Type.

## 11. Что такое Representation

### Технически

Representation — сохранённая конфигурация конкретного View.

### По-человечески

Representation — это настройка отображения.

Например пользователь открыл Table View и настроил:

- какие столбцы показывать;
- какие скрыть;
- как отсортировать;
- какой фильтр применить;
- в каком порядке поставить столбцы.

Это и есть Representation.

## 12. Representation НЕ является View

### View

- Таблица;
- Дерево;
- Канбан.

### Representation

- «Мои активные проекты»;
- «Просроченные задачи»;
- «Документы на согласовании»;
- «Скрыть системные поля».

## 13. Что хранит Representation

Representation может хранить:

- filters;
- sorting;
- hidden columns;
- column order;
- column widths;
- grouping;
- pinned columns;
- display settings;
- quick filters;
- user preferences.

## 14. Пример Representation

Entity: Проекты.

View Type: Table.

Representation: «Активные проекты».

### Настройки

- показывать только проекты со статусом «В работе»;
- скрыть системные поля;
- отсортировать по сроку;
- первым столбцом поставить «Ответственный».

## 15. Что такое View Session

### Технически

View Session — временное состояние открытого View.

### По-человечески

View Session — это то, что пользователь сейчас изменил на экране, но ещё не сохранил.

Например:

- скрыл колонку;
- передвинул колонку;
- применил фильтр;
- изменил сортировку.

Пока он не сохранил — это session state.

## 16. View Session и Saved Representation

View Session — временное состояние.

Saved Representation — сохранённая настройка.

Если пользователь изменил View Session, но не сохранил Representation, изменения не должны считаться официальными.

## 17. Dirty View State

Dirty View State — это View Session, которая отличается от сохранённой Representation.

### По-человечески

«Пользователь изменил вид отображения, но ещё не сохранил эти изменения.»

## 18. Save Flow для Representation

### Если пользователь в режиме «Все»

Режим «Все» — базовое отображение без сохранённой Representation.

Если пользователь изменил:

- видимость колонок;
- порядок колонок;
- фильтр;
- сортировку;

то при сохранении должна появиться логика:

«Сохранить как новое представление».

Пользователь сам задаёт название.

Автоматически создавать Representation с названием «Новое представление» запрещено.

### Если пользователь в существующей Representation

Если пользователь изменил существующую Representation, то при сохранении:

- обновляется текущая Representation;
- новая Representation не создаётся автоматически.

## 19. Discard Flow

Discard — отказ от сохранения изменений.

### По-человечески

«Не сохранять изменения отображения.»

После discard:

- dirty state сбрасывается;
- сохранённая Representation не меняется;
- новая Representation не создаётся.

## 20. Standalone View

Standalone View — View, открытый как отдельная страница.

Например:

- отдельная страница таблицы;
- отдельный реестр;
- отдельная рабочая область объекта.

## 21. Embedded View

Embedded View — View, встроенный в страницу как Block.

Например:

- таблица внутри страницы проекта;
- список документов внутри dashboard;
- дерево структуры внутри рабочей области.

## 22. Standalone и Embedded должны использовать одну модель

Один и тот же View Engine должен обслуживать:

- standalone view;
- embedded view.

Разница только в контейнере отображения.

## 23. Что такое Block в контексте View

Block — это место на странице, где отображается View.

Block может содержать:

- Table View;
- Tree View;
- Dashboard View;
- AI Widget;
- Entity Card;
- Text;
- File Library.

Block НЕ является View.

Block НЕ является Entity.

Block — это контейнер отображения.

## 24. Что такое Layout в контексте View

Layout отвечает за:

- размер Block;
- положение Block;
- порядок Block;
- resize;
- drag/drop.

Layout не должен управлять business data.

## 25. Composite View

Composite View — составное представление.

### По-человечески

Это экран, где несколько View объединены вместе.

Например:

- слева Table View;
- справа Entity Card;
- снизу Comments;
- сверху KPI.

Composite View НЕ должен быть хаотичной страницей.

Он должен быть управляемой композицией View.

## 26. Universal Table в новой модели

Universal Table должна стать:

Table View Engine.

### Она должна отвечать только за

- отображение данных таблицей;
- работу со столбцами;
- сортировку;
- фильтры;
- selection;
- inline editing;
- scroll;
- virtualization.

### Universal Table НЕ должна быть

- источником бизнес-данных;
- главным хранилищем сущностей;
- местом бизнес-логики.

## 27. Что относится к Table View State

Table View State:

- active filters;
- active sorting;
- visible columns;
- column order;
- column widths;
- row selection;
- scroll position;
- inline edit state.

## 28. Что относится к Table Representation

Table Representation:

- saved filters;
- saved sorting;
- saved hidden columns;
- saved column order;
- saved column widths;
- saved grouping.

## 29. Что нельзя смешивать

Запрещено смешивать:

- View Type и Representation;
- Block и View;
- View и Entity;
- Layout State и View State;
- Runtime State и Saved State.

## 30. Кто владеет View State

View State должен иметь одного владельца.

Для Table View владельцем должен быть Table View Session.

Section, Block, Toolbar и Page не должны владеть Table View State.

Они могут только читать его или вызывать явные действия.

## 31. Правильный flow изменения View

Пользователь изменил View → View Session обновилась → Dirty = true → Пользователь сохраняет → Representation сохраняется → Dirty = false → UI обновляется

## 32. Неправильный flow

### Запрещено

- менять View State через скрытые глобальные события;
- хранить одну настройку в нескольких местах;
- сохранять Representation автоматически без имени;
- изменять вкладки View при изменении Representation;
- смешивать Tab и Representation.

## 33. Минимальный checklist View Engine

После изменений View Engine всегда проверять:

### TABLE

- скрытие колонок;
- показ колонок;
- перемещение колонок;
- сортировка;
- фильтр;
- сохранение Representation;
- создание новой Representation из режима «Все»;
- обновление существующей Representation;
- F5;
- переход по левому меню.

### EMBEDDED

- таблица внутри страницы;
- таблица как Block;
- несколько таблиц на странице.

### LAYOUT

- resize блока;
- resize секции;
- drag/drop блока.

## 34. Главный принцип разработки View Engine

View Engine должен быть:

- предсказуемым;
- независимым от Page;
- независимым от Block;
- совместимым с Entity Layer;
- готовым к новым View Type.

## 35. Финальная формула

Entity хранит данные.

View показывает данные.

Representation хранит настройки View.

Block размещает View на странице.

Layout управляет пространством.

Если эти роли смешиваются — платформа становится нестабильной.
