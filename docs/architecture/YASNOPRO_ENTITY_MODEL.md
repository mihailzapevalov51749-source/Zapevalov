# YASNOPRO ENTITY MODEL

## 1. Назначение документа

Документ описывает модель цифровых объектов платформы ЯсноПро.

### Цель документа

- определить из чего состоит цифровой объект компании;
- отделить данные от отображения;
- создать единый язык платформы;
- подготовить фундамент для AI-context architecture;
- устранить хаос сущностей и hidden dependencies.

### Простыми словами

Этот документ отвечает на вопрос:

> «Как устроен цифровой объект компании внутри ЯсноПро?»

---

# 2. Главная идея платформы

В ЯсноПро всё строится вокруг объектов бизнеса.

## Не вокруг

- экранов;
- таблиц;
- модулей;
- страниц.

## А вокруг

- Проектов;
- Договоров;
- Сотрудников;
- Документов;
- Рисков;
- Задач;
- Процессов.

Каждый объект имеет:

- структуру;
- поля;
- связи;
- историю;
- представления.

---

# 3. Что такое ObjectType

## Технически

ObjectType — тип бизнес-объекта.

## По-человечески

ObjectType — это шаблон цифрового объекта.

## Например

- Проект
- Договор
- Подрядчик
- Риск
- Задача

---

# 4. Структура ObjectType

| Поле | Простое объяснение |
|---|---|
| id | внутренний идентификатор |
| key | системное имя |
| name | название объекта |
| description | описание объекта |
| icon | иконка |
| color | цвет объекта |
| fields | список полей |
| relations | список связей |
| views | доступные представления |
| permissions | права доступа |
| behaviors | поведение объекта |

---

# 5. Пример ObjectType

## ObjectType

### Проект

### Поля

- Название
- Статус
- Бюджет
- Ответственный
- Дата начала
- Дата окончания

### Связи

- Подрядчики
- Документы
- Риски
- Задачи

### Представления

- Таблица
- Канбан
- Гант
- Карточка

---

# 6. Что такое Entity

## Технически

Entity — экземпляр ObjectType.

## По-человечески

Entity — конкретный объект компании.

## Например

ObjectType: Проект  
Entity: Строительство склада №145

---

# 7. Структура Entity

| Поле | Простое объяснение |
|---|---|
| id | идентификатор объекта |
| object_type_id | к какому типу относится |
| title | название объекта |
| values | значения полей |
| relations | связи |
| metadata | системная информация |
| created_at | дата создания |
| updated_at | дата изменения |
| created_by | кто создал |
| history | история изменений |

---

# 8. Главный принцип Entity

Entity НЕ зависит от:

- таблицы;
- страницы;
- UI;
- layout.

Один и тот же Entity может одновременно:

- отображаться в таблице;
- участвовать в канбане;
- открываться карточкой;
- использоваться AI;
- участвовать в workflow.

---

# 9. Что такое Field

## Технически

Field — единица metadata объекта.

## По-человечески

Field — параметр объекта.

## Например у проекта

- Статус
- Бюджет
- Ответственный
- Срок

---

# 10. Структура Field

| Поле | Простое объяснение |
|---|---|
| id | идентификатор поля |
| key | системное имя |
| name | название поля |
| type | тип поля |
| config | настройки |
| validation | правила проверки |
| ui | настройки отображения |
| behavior | поведение поля |

---

# 11. Типы полей

## Базовые типы

| Тип | Что означает |
|---|---|
| text | короткий текст |
| long_text | большой текст |
| number | число |
| boolean | да/нет |
| date | дата |
| datetime | дата и время |
| choice | выбор |
| multi_choice | множественный выбор |
| user | пользователь |
| users | несколько пользователей |
| relation | связь |
| relations | несколько связей |
| file | файл |
| files | несколько файлов |
| lookup | вычисляемое поле |

---

# 12. Главный принцип Field

Поля должны быть reusable.

## Например

- Статус;
- Ответственный;
- Приоритет.

не должны каждый раз создаваться заново.

---

# 13. Что такое Relation

## Технически

Relation — связь между объектами.

## По-человечески

Relation показывает:

> «Как один объект связан с другим»

---

# 14. Примеры Relation

| Связь | Объяснение |
|---|---|
| Проект → Подрядчик | подрядчик участвует в проекте |
| Проект → Документ | документ относится к проекту |
| Риск → Проект | риск связан с проектом |
| Сотрудник → Задача | сотрудник отвечает за задачу |

---

# 15. Зачем нужен Relation Layer

Связи создают цифровой граф компании.

Благодаря этому:

- AI понимает контекст;
- можно анализировать зависимости;
- можно видеть impact изменений;
- можно строить аналитику;
- можно искать причины проблем.

---

# 16. Что такое View

## Технически

View — способ отображения данных.

## По-человечески

View — способ посмотреть на объект.

---

# 17. Примеры View

| View | Простое объяснение |
|---|---|
| Table | таблица |
| Tree | дерево |
| Kanban | карточки по статусам |
| Gantt | план-график |
| Calendar | календарь |
| Card | карточка объекта |
| Composite | составное отображение |

---

# 18. Главный принцип View

View НЕ хранит данные.

View использует Entity layer.

View — только отображение.

---

# 19. Что такое Representation

## Технически

Representation — конфигурация View.

## По-человечески

Representation — персональная настройка отображения.

---

# 20. Пример Representation

Есть таблица проектов.

Пользователь:

- скрыл часть столбцов;
- отсортировал данные;
- применил фильтр;
- изменил порядок столбцов.

Это Representation.

---

# 21. Representation содержит

| Настройка | Простое объяснение |
|---|---|
| filters | фильтры |
| sorting | сортировка |
| hidden_columns | скрытые столбцы |
| column_order | порядок столбцов |
| column_widths | размеры столбцов |
| grouping | группировка |
| pinned | закрепления |

---

# 22. Важное разделение

## View ≠ Representation

### View

- Таблица
- Дерево
- Канбан

### Representation

- настройки конкретного View.

---

# 23. Что такое Block

## Технически

Block — UI unit страницы.

## По-человечески

Block — отдельный визуальный элемент страницы.

## Например

- таблица;
- текст;
- AI-помощник;
- график;
- карточка;
- dashboard.

---

# 24. Block НЕ является данными

Block — это только отображение.

Block не должен быть source of truth.

---

# 25. Что такое Layout

## Технически

Layout — пространственная организация блоков.

## По-человечески

Layout отвечает:

- где находится блок;
- какого он размера;
- как блоки расположены.

---

# 26. Source of Truth

Главный принцип платформы:

## Источник истины

- ObjectType;
- Entity;
- Relation.

## НЕ

- таблица;
- UI;
- страница;
- блок.

---

# 27. Entity Graph

Все объекты платформы образуют единый граф.

## Например

Проект → связан с подрядчиком → связан с договором → связан с документами → связан с рисками → связан с задачами

Это и есть цифровая модель компании.

---

# 28. AI Context

AI использует:

- entity graph;
- relations;
- metadata;
- history;
- events.

Благодаря этому AI понимает:

- что происходит;
- что с чем связано;
- где риски;
- где зависимости;
- где отклонения.

---

# 29. Что нельзя делать

Запрещено:

- делать таблицу источником истины;
- привязывать данные к UI;
- смешивать Entity и Block;
- хардкодить бизнес-сущности;
- создавать hidden synchronization.

---

# 30. Главная цель Entity Model

Entity Model нужна для того, чтобы:

- платформа стала универсальной;
- AI понимал компанию;
- любые feature строились поверх platform core;
- система могла масштабироваться без хаоса.

---
