# YASNOPRO RELATION ENGINE MODEL

## 1. Назначение документа

Документ описывает Relation Engine платформы ЯсноПро.

### Цель документа

- определить, как объекты компании связываются между собой;
- создать основу цифрового графа компании;
- подготовить платформу к AI-context architecture;
- обеспечить dependency analysis и impact analysis;
- отделить простые ссылки от смысловых связей.

### Простыми словами

Документ отвечает на вопрос:

«Как ЯсноПро понимает, что с чем связано в компании?»

## 2. Главная идея Relation Engine

Relation Engine — это движок связей между объектами.

### По-человечески

Relation Engine показывает, как разные части компании связаны между собой.

Например:

- проект связан с подрядчиком;
- договор связан с проектом;
- задача зависит от другой задачи;
- документ относится к этапу;
- риск влияет на срок.

## 3. Почему Relation Engine важен

Без связей платформа видит только отдельные объекты.

Со связями платформа видит систему.

### Без Relation Engine

- есть отдельные карточки;
- есть отдельные таблицы;
- есть отдельные документы.

### С Relation Engine

- появляется цифровой граф компании;
- AI понимает контекст;
- можно анализировать зависимости;
- можно видеть последствия изменений.

## 4. Что такое Relation

### Технически

Relation — связь между двумя объектами.

### По-человечески

Relation — это смысловая связь:

«Один объект связан с другим определённым образом.»

## 5. Пример Relation

- Проект → имеет подрядчика → Подрядчик
- Задача → зависит от → Задача
- Документ → относится к → Проект
- Риск → влияет на → Срок

## 6. Базовая структура Relation

Relation состоит из трёх главных частей:

- from
- relation
- to

### По-человечески

| Часть | Значение |
|---|---|
| from | от какого объекта идёт связь |
| relation | какой тип связи |
| to | к какому объекту идёт связь |

## 7. Пример from / relation / to

| from | relation | to |
|---|---|---|
| Проект | имеет договор | Договор |
| Подрядчик | выполняет работы по | Проект |
| Задача | зависит от | Задача |
| Документ | относится к | Объект строительства |

## 8. Что такое Relation Type

### Технически

Relation Type — тип связи.

### По-человечески

Relation Type отвечает на вопрос:

«Каким образом эти объекты связаны?»

## 9. Примеры Relation Type

| Relation Type | Простое объяснение |
|---|---|
| belongs_to | относится к |
| has_many | имеет много |
| depends_on | зависит от |
| responsible_for | отвечает за |
| participates_in | участвует в |
| affects | влияет на |
| blocks | блокирует |
| approved_by | согласовано кем |
| created_by | создано кем |

## 10. Что такое Relation Instance

### Технически

Relation Instance — конкретная связь между конкретными Entity.

### По-человечески

Это уже реальная связь в компании.

Например:

Проект «Склад №1» связан с подрядчиком «СтройМонтаж».

## 11. Relation Definition vs Relation Instance

Relation Definition — правило связи.

Relation Instance — конкретная связь.

### Пример

#### Relation Definition

Проект может иметь подрядчиков.

#### Relation Instance

Проект «Склад №1» имеет подрядчика «СтройМонтаж».

## 12. Cardinality

### Технически

Cardinality — количество связей между объектами.

### По-человечески

Cardinality отвечает:

«Сколько объектов может быть связано?»

## 13. Типы Cardinality

| Тип | Простое объяснение | Пример |
|---|---|---|
| one-to-one | один к одному | сотрудник → профиль |
| one-to-many | один ко многим | проект → задачи |
| many-to-one | многие к одному | задачи → проект |
| many-to-many | многие ко многим | сотрудники ↔ проекты |

## 14. Direction

### Технически

Direction — направление связи.

### По-человечески

Direction показывает, откуда и куда идёт связь.

Например:

- Проект → имеет → Задачи
- Задача → относится к → Проект

Это две стороны одной смысловой связи.

## 15. Bidirectional Relations

Связь может быть двусторонней.

Например:

- Проект имеет задачи.
- Задача относится к проекту.

Платформа должна уметь показывать обе стороны связи.

## 16. Semantic Relation

### Технически

Semantic Relation — смысловая связь.

### По-человечески

Это связь, смысл которой понятен человеку и AI.

Например:

Не просто:

relation_id = 14

А:

Подрядчик выполняет работы по проекту.

## 17. Почему Semantic Relation важен для AI

AI должен понимать не только факт связи, но и её смысл.

Например:

Если задача зависит от другой задачи, AI понимает:

- нельзя завершить задачу B до задачи A;
- просрочка A влияет на B;
- риск может перейти выше по цепочке.

## 18. Relation Graph

### Технически

Relation Graph — граф связей объектов.

### По-человечески

Relation Graph — карта связей компании.

Она показывает:

- кто с кем связан;
- что от чего зависит;
- какие объекты влияют друг на друга.

## 19. Пример Relation Graph

Проект → Договор → Подрядчик → Документы → Задачи → Риски

Это не просто список.

Это сеть связей.

## 20. Relation Engine и AI

Relation Engine является основой AI-context.

AI использует связи, чтобы понимать:

- контекст объекта;
- причины проблем;
- зависимости;
- последствия;
- риски;
- ответственных.

## 21. Dependency Relation

Dependency Relation — связь зависимости.

### По-человечески

«Один объект зависит от другого.»

### Пример

Задача B зависит от задачи A.

Если задача A просрочена, задача B тоже под риском.

## 22. Impact Analysis

### Технически

Impact Analysis — анализ влияния изменений.

### По-человечески

Impact Analysis отвечает:

«Если изменится этот объект, что ещё пострадает?»

## 23. Пример Impact Analysis

Если подрядчик задерживает работы:

- какие проекты пострадают?
- какие задачи сдвинутся?
- какие договоры под риском?
- какие документы нужно пересмотреть?

## 24. Graph Traversal

### Технически

Graph Traversal — обход графа связей.

### По-человечески

Это когда система идёт от одного объекта по связям к другим объектам.

Например:

Покажи всё, что связано с проектом, на глубину 3.

## 25. Relation Permissions

Не все связи должны быть видны всем пользователям.

Relation Engine должен учитывать permissions.

Например:

- финансовые связи видит только финансовый отдел;
- кадровые связи видит HR;
- договорные связи видят юристы.

## 26. Runtime Relations

Runtime Relations — связи, с которыми работают сотрудники.

### Примеры

- прикрепить документ к проекту;
- связать задачу с риском;
- назначить ответственного;
- указать зависимость задач.

## 27. Designer Relations

Designer Relations — настройка правил связей.

### Примеры

- проект может иметь много задач;
- договор относится к одному подрядчику;
- риск может быть связан с несколькими объектами;
- задача может зависеть от другой задачи.

## 28. Runtime vs Designer в Relation Engine

Designer создаёт типы и правила связей.

Runtime использует конкретные связи в работе.

## 29. Relation и Field

Relation может отображаться как Field.

Например:

Поле «Подрядчик» в проекте может быть relation к объекту «Подрядчик».

Но внутри платформы это не просто текстовое поле.

Это связь между Entity.

## 30. Lookup и Relation

Lookup — это отображение данных из связанного объекта.

Например:

У проекта есть подрядчик.

Через lookup можно показать ИНН подрядчика в таблице проектов.

Lookup не является самостоятельной связью.

Lookup использует Relation.

## 31. Relation и View

View может показывать связи.

Например:

- в таблице показать связанные документы;
- в карточке показать связанные задачи;
- в графе показать сеть связей;
- в дереве показать иерархию.

## 32. Relation и Event

Создание, изменение и удаление связи должно порождать событие.

### Примеры

- relation.created
- relation.updated
- relation.deleted

Это нужно для:

- истории;
- уведомлений;
- AI context;
- автоматизации.

## 33. Relation Lifecycle

Жизненный цикл связи:

1. Relation Definition создана в Designer.
2. Пользователь создаёт Relation Instance в Runtime.
3. Relation Instance используется в View.
4. Relation Instance участвует в AI context.
5. Изменение Relation Instance попадает в Event History.

## 34. Что нельзя делать

Запрещено:

- хранить связи только как текст;
- делать связи только через название объекта;
- смешивать lookup и relation;
- делать relation без смысла;
- использовать relation_id без semantic meaning;
- строить AI context без relation graph.

## 35. Минимальная структура Relation Definition

Relation Definition должна содержать:

- id;
- key;
- name;
- from_object_type;
- to_object_type;
- relation_type;
- cardinality;
- direction;
- reverse_name;
- permissions;
- metadata.

## 36. Минимальная структура Relation Instance

Relation Instance должна содержать:

- id;
- relation_definition_id;
- from_entity_id;
- to_entity_id;
- metadata;
- created_at;
- created_by;
- status.

## 37. Пример Relation Definition

### Name

Проект имеет подрядчиков

### From

Project

### Relation

has_contractor

### To

Contractor

### Cardinality

many-to-many

### Reverse

Подрядчик участвует в проектах

## 38. Пример Relation Instance

### Project

Склад №1

### Relation

имеет подрядчика

### Contractor

СтройМонтаж

## 39. Relation Engine как нервная система платформы

ObjectType и Entity создают тело платформы.

Relation Engine создаёт нервную систему.

Он связывает части компании в единую цифровую модель.

## 40. Финальная формула

Entity без Relation — отдельная карточка.

Entity с Relation — часть цифровой модели компании.

Relation Engine превращает ЯсноПро из набора объектов в связанную business platform.
