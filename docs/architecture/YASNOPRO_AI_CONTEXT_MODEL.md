# YASNOPRO AI CONTEXT MODEL

## 1. Назначение документа

Документ описывает AI Context Engine платформы ЯсноПро.

### Цель документа

- определить, как AI понимает компанию;
- создать фундамент AI-native architecture;
- превратить данные в смысловой контекст;
- подготовить платформу к AI reasoning и AI agents;
- создать основу интеллектуальной цифровой модели компании.

### Простыми словами

Документ отвечает на вопрос:

> «Как AI понимает, что происходит внутри компании?»

---

# 2. Главная идея AI Context Engine

AI Context Engine — это слой понимания платформы.

## По-человечески

AI Context Engine помогает AI видеть:

- объекты;
- связи;
- историю;
- процессы;
- роли;
- события;
- последствия;
- зависимости.

AI должен понимать не только данные, но и смысл происходящего.

---

# 3. Почему обычного AI недостаточно

## Обычный AI

- работает только с текстом;
- не знает структуру компании;
- не понимает связи объектов;
- не понимает последствия;
- не знает историю.

## AI-native платформа должна давать AI

- context;
- graph;
- semantic meaning;
- business structure.

---

# 4. Что такое AI Context

## Технически

AI Context — совокупность данных, связей, истории и смыслов, используемых AI.

## По-человечески

AI Context — это всё, что помогает AI понимать компанию как систему.

---

# 5. Что входит в AI Context

AI Context включает:

- Entity Graph;
- Relation Graph;
- Event Timeline;
- Metadata;
- Documents;
- Comments;
- Workflow;
- Permissions;
- Runtime Context;
- User Role;
- Business Structure.

---

# 6. Entity Context

AI должен понимать:

- что это за объект;
- к какому ObjectType относится;
- какие у него поля;
- какие значения важны.

## Пример

### Проект

- статус;
- срок;
- бюджет;
- ответственный.

---

# 7. Relation Context

AI должен понимать:

- с чем связан объект;
- какие зависимости существуют;
- какие объекты влияют друг на друга.

## Пример

Проект связан:

- с подрядчиком;
- с договором;
- с рисками;
- с задачами.

---

# 8. Event Context

AI должен понимать историю событий.

## Например

- кто изменил статус;
- когда возник риск;
- когда документ ушёл на согласование;
- когда появилась просрочка.

---

# 9. Semantic Context

## Технически

Semantic Context — смысловое понимание данных.

## По-человечески

AI должен понимать смысл связей и действий.

## Например

Не просто:

relation_id = 14

А:

> «Подрядчик выполняет работы по проекту.»

---

# 10. Почему Semantic Context критичен

Без semantic layer AI видит только таблицы и ID.

С semantic layer AI понимает:

- процессы;
- зависимости;
- риски;
- роли;
- причинно-следственные связи.

---

# 11. Context Graph

## Технически

Context Graph — граф связей AI-context.

## По-человечески

Это цифровая карта компании, которую использует AI.

Она показывает:

- объекты;
- связи;
- события;
- влияние;
- зависимости.

---

# 12. Context Window

## Технически

Context Window — объём контекста, доступный AI.

## По-человечески

Context Window отвечает:

> «Что AI должен учитывать прямо сейчас?»

---

# 13. Что может входить в Context Window

Например:

- текущий объект;
- связанные объекты;
- последние события;
- comments;
- workflow state;
- permissions;
- role пользователя.

---

# 14. Runtime Context

Runtime Context — контекст текущей работы пользователя.

## Например

- пользователь открыл проект;
- смотрит задачи;
- обсуждает риск;
- работает в kanban.

AI должен учитывать:

- где пользователь находится;
- с чем работает;
- что пытается сделать.

---

# 15. Designer Context

Designer Context — контекст моделирования платформы.

## Например

- аналитик создаёт ObjectType;
- проектирует relation;
- настраивает workflow;
- создаёт representation.

AI должен помогать проектировать систему.

---

# 16. Organizational Context

AI должен понимать структуру компании.

## Например

- подразделения;
- роли;
- зоны ответственности;
- hierarchy;
- ownership.

---

# 17. Permission Context

AI должен учитывать permissions.

AI не должен:

- показывать скрытые данные;
- раскрывать restricted relations;
- обходить security model.

---

# 18. AI Memory

## Технически

AI Memory — память AI о данных и взаимодействиях.

## По-человечески

AI должен помнить:

- что происходило;
- какие паттерны были раньше;
- какие решения уже принимались.

---

# 19. Типы AI Memory

| Тип | Простое объяснение |
|---|---|
| Short-term | текущий контекст |
| Working Memory | активная задача |
| Organizational Memory | знания компании |
| Historical Memory | история изменений |
| Semantic Memory | понимание структуры |

---

# 20. AI Reasoning

## Технически

AI Reasoning — механизм логических выводов AI.

## По-человечески

AI Reasoning отвечает:

> «Как AI делает выводы?»

---

# 21. Пример AI Reasoning

AI должен понимать:

Не просто:

> «Проект просрочен.»

А:

- подрядчик задержал документы;
- согласование длилось 8 дней;
- монтаж не стартовал;
- риск не был обработан;
- проект вышел за срок.

---

# 22. Dependency Reasoning

AI должен анализировать зависимости.

## Например

- задача зависит от другой задачи;
- договор зависит от согласования;
- риск влияет на сроки.

---

# 23. Pattern Recognition

AI должен видеть повторяющиеся patterns.

## Например

- одинаковые причины просрочек;
- типовые проблемы подрядчиков;
- повторяющиеся ошибки согласования.

---

# 24. AI Recommendations

AI должен уметь предлагать рекомендации.

## Например

- предупредить о риске;
- предложить действие;
- указать узкое место;
- показать зависимость.

---

# 25. Пример AI Recommendation

> «У проекта высокий риск просрочки. Причина: подрядчик задерживает согласование третий раз за месяц.»

---

# 26. AI Explanation

AI должен объяснять выводы.

AI не должен быть «чёрным ящиком».

Пользователь должен понимать:

- почему AI сделал вывод;
- на каких данных основано решение;
- какие события и связи использовались.

---

# 27. Explainable AI

## Технически

Explainable AI — объяснимый AI.

## По-человечески

AI должен уметь показать:

> «Я сделал этот вывод потому что…»

---

# 28. AI Agents

AI Agent — специализированный AI внутри платформы.

---

# 29. Примеры AI Agents

| Agent | Что делает |
|---|---|
| Project AI | анализирует проекты |
| Risk AI | анализирует риски |
| Workflow AI | анализирует процессы |
| Executive AI | помогает руководителю |
| Analyst AI | помогает аналитикам |
| Document AI | работает с документами |

---

# 30. Runtime AI

Runtime AI помогает сотрудникам.

## Например

- искать информацию;
- анализировать объекты;
- объяснять связи;
- подсказывать действия.

---

# 31. Designer AI

Designer AI помогает моделировать платформу.

## Например

- предлагать ObjectType;
- искать дубли;
- оптимизировать relations;
- выявлять complexity.

---

# 32. AI Context и View

AI должен понимать:

- какой View открыт;
- что пользователь сейчас анализирует;
- какие Representation используются.

## Например

Пользователь открыл:

> «Просроченные задачи».

AI должен понимать, что пользователь анализирует проблемы.

---

# 33. AI Context и Comments

AI должен учитывать:

- обсуждения;
- mentions;
- решения;
- договорённости.

Comments — часть organizational memory.

---

# 34. AI Context и Documents

AI должен понимать:

- документы;
- версии;
- согласования;
- связанные объекты.

Документ не должен быть «файлом без контекста».

---

# 35. AI Context и Workflow

AI должен понимать:

- этап процесса;
- текущий workflow state;
- блокировки;
- approvals;
- dependencies.

---

# 36. AI Safety

AI должен соблюдать:

- permissions;
- privacy;
- access control;
- explainability;
- traceability.

---

# 37. AI Traceability

## Технически

Traceability — возможность проследить ход reasoning AI.

## По-человечески

Пользователь должен понимать:

- почему AI дал рекомендацию;
- какие данные использовались;
- какие связи анализировались.

---

# 38. AI и Event Timeline

AI должен анализировать:

- последовательность событий;
- timeline объекта;
- динамику изменений.

AI reasoning без Event Timeline считается неполным.

---

# 39. AI Context Lifecycle

AI Context формируется:

1. Entity Layer
2. Relation Graph
3. Event Timeline
4. Permissions
5. Runtime Context
6. Semantic Layer
7. Organizational Memory

---

# 40. Что нельзя делать

Запрещено:

- строить AI только на текстовых запросах;
- игнорировать relation graph;
- игнорировать event history;
- давать AI доступ без permissions;
- использовать AI без explainability.

---

# 41. AI Context как главное отличие AOBP

## Большинство систем

- хранят данные;
- показывают данные.

## AOBP должна

- понимать компанию;
- понимать связи;
- понимать последствия;
- понимать процессы.

---

# 42. AI Context как цифровое мышление компании

- Entity Layer хранит объекты.
- Relation Engine хранит связи.
- Event Engine хранит историю.
- AI Context Engine превращает всё это в понимание.

---

# 43. Финальная формула

AI Context Engine отвечает за то, чтобы AI видел:

> не отдельные таблицы, а живую цифровую систему компании.

Именно AI Context превращает ЯсноПро из конструктора данных в AI-native Object-centric Business Platform.
